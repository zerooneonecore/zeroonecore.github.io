Dealing with bytes

The goal here is to develop some facilities to make working with bytes easier, to facilitate the creation of an ELF file from the structs defined within the header files.
The first challenge here is to calculate accurately the size of a partial struct, where a partial struct is a struct, including its first member, truncated to some subsequent member. This may appear to be something of a non sequitur, but if we needed to calculate the size of a 'full struct', the sizeof() function can return this trivially. If the struct is padded to an even byte multiple, then this isn't so difficult, so the task is to determine if a struct is padded. I first explored whether this was necessary, and found examples where structs were indeed padded by the compiler without special settings but could be padded to different multiples, e.g, 4 or 8. Which means given sizeof(A) = 16 for example, we cannot make the assumption as to how many members A it has, as there are different possibilities, for example,

16 = 4 * 4 = 8 * 2 = 8 + 4 + 4


Fortunately, or well, theoretically we don't actually need to deal with a 'partial struct', because the section header table and program header table each are their own struct as per man elf, hence as long as offsets are calculated to a struct and not its subsequent members, the sizeof() function should work wonderfully for calculating the offsets. The ELF header table has been more or less covered, the next order of business is the program header table. The two problematic fields are p_vaddr, and p_offset. The man elf entry does not provide much information as to what constitutes a virtual address, so we may have to consult the ELF specification for this. The ELF specification did not provide much actionable or at least obvious information about p_vaddr, so I decided the easiest way to figure this out would be to compile different types of executables, and study the program header tables of them. The first program I would compile would be one with hardcoded addresses, referred to as a 'statically linked program'. I used the objdump utility rather than readelf because I found the output more intelligible.

Program Header:
LOAD off 0x0000000000000000 vaddr 0x0000000000400000 paddr 0x0000000000400000 align 2**12
